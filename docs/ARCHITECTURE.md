# Architecture

## Overview
This project is split into portable core logic and ESP-IDF-specific integration:
- `components/core_sm`: framework-agnostic FSM/HSM engines
- `components/core_blinky`: framework-agnostic blinky model/menu logic
- `components/blinky_idf`: ESP-IDF + hardware integration (button, LED PWM, app state machine)

The app entry point (`main/main.c`) initializes the LED state machine and steps it in a loop.

## Decoupling Refactor
Goal: isolate reusable logic from ESP-IDF/HAL code so core behavior can be tested and reused independently.

Migration map:
- `components/blinky/fsm_engine.h` -> `components/core_sm/fsm_engine.h`
- `components/blinky/led_model.*` + `menu_logic.*` -> `components/core_blinky/`
- `components/blinky/button.*` + `led_sm.*` + Kconfig integration -> `components/blinky_idf/`
- Unit tests moved with ownership:
  - `core_sm/test/*`
  - `core_blinky/test/*`
  - `blinky_idf/test/*`

## Runtime Behavior
- LED brightness follows a waveform generated by the state machine.
- Button input drives state transitions.
- Controls:
  - Short press: pause/resume waveform.
  - Long press (~3 s): enter/exit wave selection menu.
  - In menu: short press cycles wave type (square, saw up/down, triangle, sine).

Default pins:
- LED output: `GPIO_NUM_14` (PWM, active-low, 5 kHz)
- Button input: `GPIO_NUM_9` (active-low with pull-up)

## Configuration
Menuconfig path: `Component config -> Blinky` (from `components/blinky_idf/Kconfig`).

Key knobs:
- LED GPIO, button GPIO, active-low setting
- Poll interval, debounce count, long-press duration
- Wave period, sine LUT max steps, saw/triangle step size
- PWM frequency
- Startup waveform
- Boot pattern enable and pulse duration
- Intensity logging enable/disable

## Testing Strategy
Tests are Unity-based and split by ownership:
- `components/core_sm/test/test_fsm_engine.c`
- `components/core_sm/test/test_hsm_engine.c`
- `components/core_blinky/test/test_led_model.c`
- `components/core_blinky/test/test_menu_logic.c`
- `components/blinky_idf/test/test_button.c`

Unit tests run via ESP-IDF Unit Test App with this repo injected through `EXTRA_COMPONENT_DIRS`.

## Repo Hygiene
- `unity-app/` is local scratch/test harness and intentionally git-ignored.

## Next Intention: Event-Driven Transition
The next architectural step is moving from the current polling/step-loop orchestration
to an event-driven model.

Planned direction:
- Use the HSM in `core_sm` as the primary orchestration mechanism.
- Define explicit event contracts at module boundaries (button, timer tick, menu actions, control events).
- Route events through a queue/dispatcher layer in `blinky_idf`, while keeping core logic portable.
- Preserve current behavior parity while migrating (existing tests remain the guardrail).
