# blinky_c6

ESP-IDF app for ESP32-C6 that drives an LED using a waveform-based state machine. A single button controls run/pause and a wave-selection menu. The LED can display square, saw, triangle, or sine patterns, with configurable timing via `menuconfig`. Unit tests are written with Unity and run through Espressif’s Unit Test App.

**Blinky Behavior**
- LED brightness follows a waveform generated by the state machine (e.g., sine).
- Button input drives state transitions (see `components/blinky/button.c`).
- The app entry point simply initializes the LED state machine and steps it in a loop (`main/main.c`).

**How To Interact (Default Pins)**
- LED output: `GPIO_NUM_14` (PWM, active-low, 5 kHz).
- Button input: `GPIO_NUM_9` (active-low with pull-up).

**Controls**
- Short press:
  Pause or resume the waveform.
- Long press (~3 s):
  Enter/exit the wave selection menu.
- While in menu:
  Short press cycles wave type (square, saw up/down, triangle, sine). The LED
  continues animating using the currently selected wave so it can be observed
  without a serial monitor.

The app also logs state transitions and wave names to the serial console.
Pins and timings are defined in `components/blinky/led_sm.c` if you need to change them.

**Configuration (menuconfig)**
The blinky component exposes config options under `Component config → Blinky`:
- LED GPIO, Button GPIO, active-low setting
- Poll interval, debounce count, long-press duration
- Wave period (target cycle time)
- Sine LUT size is derived at build time from period/poll (minimum 8, capped by `BLINKY_SINE_STEPS_MAX`)
- Saw/triangle step size and PWM frequency

Note: Very long periods combined with very short poll intervals increase the
LUT size and memory usage up to `BLINKY_SINE_STEPS_MAX`.
- Startup waveform selection
- Boot pattern enable and pulse duration (LED flashes N times at boot to indicate the startup wave)
- Intensity logging (enable/disable per-update LED percentage logs)

- `build/` main app build output (generated)

**Requirements**
- ESP-IDF installed and activated (e.g. `export.sh` on Linux/macOS)
- A configured target for ESP32-C6

**Configure Target**
```bash
idf.py set-target esp32c6
```
VS Code tasks use the `idf.target` setting in `.vscode/settings.json`. To change
targets for tasks, update that value (for example, `esp32c6` → `esp32s3`) and
re-run the appropriate `Set Target` task.

**Build**
```bash
idf.py -D CCACHE_ENABLE=1 build
```

**Flash and Monitor**
```bash
idf.py -p /dev/ttyUSB0 flash monitor
```
Replace the port with your device (for example `/dev/ttyACM0` on Linux or `COMx` on Windows).
VS Code tasks use the `idf.port` (and `idf.monitorPort`) values in `.vscode/settings.json`.
Update those settings if your serial port changes.

**Run Unit Tests (Unity via ESP-IDF Unit Test App)**
ESP-IDF runs Unity tests using its Unit Test App. Our tests live in
`components/blinky/test/`, so we build the Unit Test App and point it at
our components directory with `EXTRA_COMPONENT_DIRS`.

Option A (terminal, explicit project dir):
```bash
idf.py -C $IDF_PATH/tools/unit-test-app -B $PWD/build/unit-test-app -D EXTRA_COMPONENT_DIRS=$PWD/components set-target esp32c6
idf.py -C $IDF_PATH/tools/unit-test-app -B $PWD/build/unit-test-app -D EXTRA_COMPONENT_DIRS=$PWD/components -D SDKCONFIG=$PWD/build/unit-test-app/sdkconfig -D CCACHE_ENABLE=1 -T blinky build
idf.py -C $IDF_PATH/tools/unit-test-app -B $PWD/build/unit-test-app -D SDKCONFIG=$PWD/build/unit-test-app/sdkconfig -T blinky -p /dev/ttyACM0 flash monitor
```

Option C (VS Code Tasks, integrated in this workspace):
- Open the repo root as your workspace
- Open the VS Code menu `Terminal → Run Task...` (or use Command Palette `Tasks: Run Task`)
- Select `IDF: Set Target (tests)` (run once per target)
- Select `IDF: Build (tests)`
- Select `IDF: Flash+Monitor (tests)`

The tasks are defined in `.vscode/tasks.json` and use the port from
`.vscode/settings.json` (`idf.port`).

**Build Caching Notes**
- The app and unit-test app use separate build directories (`build/` and
  `build/unit-test-app/`), so each has its own incremental cache.
- `ccache` is shared across both builds, which helps avoid recompiling
  identical objects when switching.
- Full rebuilds are only needed when changing targets, toolchains/IDF
  versions, or after running `fullclean`.

**VS Code (ESP-IDF Extension)**
- Command Palette (`Ctrl+Shift+P`) → `ESP-IDF: Build Project`
- Command Palette (`Ctrl+Shift+P`) → `ESP-IDF: Unit Test`

If IntelliSense squiggles appear, reconfigure the project so `compile_commands.json` is regenerated.
